Компилятор Lisp
==============================

Лексический анализатор
----------------------

Функция ``lisp-lexer`` в ``lexer.lsp`` отвечает за преобразование входной строки в список лексем. Она пропускает пробельные символы и другие разделители между лексемами и использует комбинаторы парсеров для распознавания различных типов лексем.

Функция ``parse-token`` является основной точкой входа для распознавания отдельных лексем. Она последовательно применяет парсеры для скобок, чисел, строк, символов и других конструкций к текущему потоку ввода.

После специальных символов, таких как `'`, `` ` ``, `,`, `,@`, `#`, `#'`, не допускаются пробельные символы. Для соблюдения этого правила парсеры вроде ``parse-quote-token``, ``parse-backquote-token``, ``parse-comma-token`` реализованы так, чтобы сразу после распознавания специального символа пытаться распознать следующую лексему без пропуска разделителей. Это гарантирует корректную обработку конструкций вида `'VAR` или `#'F` как единого лексического блока.

Фаза 1 — Статический анализ и компиляция исходного дерева
---------------------------------------------------------

Исходное S-выражение преобразуется в другое S-выражение, состоящее из ограниченного набора базовых операций.

- **Константы** остаются без изменений.  
- **Переменные** классифицируются как локальные, глобальные или свободные:
  - Для локальных переменных вычисляется смещение внутри текущего кадра активации.
  - Для глобальных переменных определяется индекс в списке ``*global-variables*``.
  - Для свободных переменных вычисляются два смещения: смещение кадра относительно текущего и смещение внутри целевого кадра.
  - Таким образом, все переменные разрешаются статически, без поиска во время выполнения.
- **Глобальные переменные** создаются при первом присваивании и сохраняются в списке ``*global-variables*``.
- **Последовательности выражений** (``progn``) компилируются в форму ``SEQ``. Для единственного выражения форма ``progn`` устраняется.
- **Условная форма ``if``** компилируется в ``ALTER``, куда передаются скомпилированные аргументы: условие, ветка «истина», ветка «ложь».
- **Присваивания** анализируются аналогично обращениям к переменным: определяется тип переменной и компилируется выражение правой части.
- **Пользовательские функции** с фиксированным числом аргументов компилируются как лямбда-выражения. Их имена, смещение окружения, число аргументов и тело сохраняются в таблице ``*fix-functions*``. Для анонимных функций генерируются уникальные имена.
- **Функции с переменным числом аргументов** сохраняются в таблице ``*nary-functions*``, вместе с числом фиксированных аргументов (остальные аргументы передаются как список).
- В **форме ``labels``** окружение локальных функций расширяется, генерируется код для тел функций, после чего окружение восстанавливается.
- **Примитивы** с фиксированным числом аргументов хранятся в ``*fix-primitives*``, с переменным — в ``*nary-primitives*``, с указанием числа аргументов.
- При **вызове функции** определяется её тип (глобальная, лямбда, примитив), проверяется соответствие числа аргументов:
  - Для лямбда-функций вставляется скомпилированное тело.
  - Для всех функций создаётся кадр активации с известным размером, активируется связанное окружение, аргументы вычисляются слева направо и помещаются в кадр.
  - Для примитивов кадр активации не создаётся.
- **Компиляция ``tagbody``**:
  - Тело преобразуется в форму ``SEQ``.
  - Символы (метки) заменяются на ``(LABEL <метка> (NOP))``.
  - Форма ``go`` компилируется в ``GOTO``.
- **Компиляция ``CATCH`` / ``THROW``**:
  - ``CATCH`` генерирует метку окончания блока и создаёт запись в стеке catch с именем метки, адресом перехода, номером кадра и позицией в стеке.
  - ``THROW`` ожидает, что метка находится в аккумуляторе, а возвращаемое значение — вычислено. Он ищет соответствующую запись в стеке catch, восстанавливает кадр, стек и переходит на сохранённый адрес.

Промежуточные операции после Фазы 1:

- ``CONST expr`` — выражение-константа.
- ``LOCAL-REF j`` — доступ к j-й локальной переменной.
- ``GLOBAL-REF i`` — доступ к i-й глобальной переменной.
- ``DEEP-REF i j`` — доступ к свободной переменной: i — смещение кадра, j — индекс в кадре.
- ``SEQ <expr list>`` — последовательное выполнение выражений.
- ``ALTER cond true false`` — условный оператор.
- ``LOCAL-SET j expr`` — присваивание локальной переменной.
- ``GLOBAL-SET i expr`` — присваивание глобальной переменной.
- ``DEEP-SET i j expr`` — присваивание свободной переменной.
- ``LABEL lab expr`` — выражение с меткой.
- ``FIX-LET num args expr`` — let-форма с фиксированным числом связываний.
- ``FIX-CLOSURE name body`` — замыкание функции с фиксированным числом аргументов.
- ``PRIM-CLOSURE name`` — замыкание примитива с фиксированным числом аргументов.
- ``NPRIM-CLOSURE name`` — замыкание примитива с переменным числом аргументов.
- ``RETURN`` — выход из функции.
- ``FIX-CALL name env args`` — вызов функции с фиксированным числом аргументов.
- ``NARY-CALL name num env args`` — вызов функции с переменным числом аргументов (num — фиксированные аргументы).
- ``FIX-PRIM name args`` — вызов примитива с фиксированным числом аргументов.
- ``NARY-PRIM name num args`` — вызов примитива с переменным числом аргументов.
- ``NOP`` — отсутствие операции.
- ``GOTO lab`` — безусловный переход.
- ``CATCH tag_expr body_expr`` — блок catch.
- ``THROW tag_expr ret_expr`` — немедленный выход из блока catch.

Фаза 2 — Оптимизация преобразованного выражения
-----------------------------------------------

**Оптимизация загрузки аккумулятора**

- В последовательностях загрузочных операций (``CONST``, ``LOCAL-REF``, ``GLOBAL-REF``, ``DEEP-REF``) все, кроме последней, заменяются на ``NOP``.
- Если сразу после ``*-SET`` следует ``*-REF`` той же переменной, то ``*-REF`` удаляется.

**Оптимизация тривиальных условий**

- ``(ALTER (CONST T) expr1 expr2)`` → ``expr1``
- ``(ALTER (CONST NIL) expr1 expr2)`` → ``expr2``

**Оптимизация арифметики**

- Примитивы с переменным числом аргументов (например, ``+``) преобразуются в цепочку бинарных вызовов:
  - ``(NARY-PRIM + 0 (a b c d))`` → ``(+ (+ (+ a b) c) d)``

**Оптимизация ``funcall`` и ``apply``**

- Устраняется избыточное дублирование аргументов в стеке при вложенных вызовах.

Фаза 3 — Генерация линейных инструкций
--------------------------------------

Дерево промежуточных форм преобразуется в линейную последовательность инструкций (ассемблер).

Инструкции после генерации:

- ``CONST val`` — загрузить константу в ACC.
- ``LOCAL-REF j`` — загрузить j-ю локальную переменную.
- ``GLOBAL-REF i`` — загрузить i-ю глобальную переменную.
- ``DEEP-REF i j`` — загрузить свободную переменную.
- ``LOCAL-SET j`` — сохранить ACC в j-ю локальную переменную.
- ``GLOBAL-SET i`` — сохранить ACC в i-ю глобальную переменную.
- ``DEEP-SET i j`` — сохранить ACC в свободную переменную.
- ``PUSH`` — поместить ACC в стек.
- ``POP`` — извлечь из стека в ACC.
- ``LABEL name`` — символическая метка (не генерирует код).
- ``JMP label`` — безусловный переход.
- ``JNT label`` — переход, если ACC == NIL.
- ``ALLOC n`` — создать кадр активации размера n, загрузить аргументы из стека (в обратном порядке).
- ``PRIM name`` — вызов примитива.
- ``PACK n`` — упаковать n верхних элементов стека в список.
- ``REG-CALL name`` — вызов функции по метке.
- ``SAVE-ENV`` — сохранить текущий кадр в стек.
- ``SET-ENV num`` — установить кадр по глубине вызова.
- ``RESTORE-ENV`` — восстановить кадр из стека.
- ``FIX-CLOSURE name`` — создать замыкание с текущим окружением и меткой функции.
- ``PRIM-CLOSURE name`` — создать замыкание примитива (фикс. аргументы).
- ``NPRIM-CLOSURE name`` — создать замыкание примитива (перем. аргументы).
- ``RETURN`` — возврат из функции.
- ``CATCH ofs`` — создать блок catch; метка в ACC, конец блока по смещению ``ofs``.
- ``THROW`` — выполнить немедленный переход на блок catch с меткой на вершине стека; результат в ACC.

Генерация происходит рекурсивно:

- ``CONST``, ``LOCAL-REF``, ``GLOBAL-REF``, ``DEEP-REF``, ``RETURN`` → соответствующие инструкции.
- ``GLOBAL-SET``, ``LOCAL-SET``, ``DEEP-SET`` → сначала вычисляется выражение, затем генерируется инструкция записи.
- ``LABEL`` → генерируется переход вокруг тела функции, затем тело, затем метка окончания.
- ``SEQ`` → генерация каждого подвыражения по порядку.
- ``ALTER`` → генерация условия, условного перехода, веток и меток.
- ``FIX-PRIM`` → вычисление и ``PUSH`` каждого аргумента, затем ``PRIM``.
- ``NARY-PRIM`` → вычисление аргументов, ``PUSH``, ``PACK`` для необязательных, затем ``PRIM``.
- ``REG-CALL`` → ``SET-ENV``, ``ALLOC``, передача аргументов, ``REG-CALL``, ``RESTORE-ENV``.

Фаза 4 — Ассемблер
-------------------

Генерация байт-кода выполняется в два прохода:

1. **Первый проход**:  
   - Мнемоники заменяются на опкоды (байты).  
   - Метки записываются в хеш-таблицу как адрес следующей инструкции.  
   - Инструкции перехода сохраняют пары (адрес, имя метки) в отдельный список.

2. **Второй проход**:  
   - Адреса переходов заменяются на реальные смещения, полученные из хеш-таблицы меток.

**Выход компилятора**:
- Список констант.
- Число глобальных переменных.
- Массив байт-кода.

Точка входа определяется как первая непереходная инструкция (после разрешения начальных ``JMP``).

Виртуальная машина
------------------------------

Точка входа — функция ``(vm-run program)``, где ``program`` — массив байт-кода.

Архитектура виртуальной машины
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Виртуальная машина включает:

- **Память программы** — массив инструкций (чисел).
- **Память констант** — массив объектов.
- **Память глобальных переменных** — массив объектов.
- **Стек данных** — хранит произвольные объекты.
- **Список кадров активации** — древовидная структура, каждый кадр — массив:
  - [0] — ссылка на предыдущий кадр,
  - [1] — глубина вызова (для быстрого доступа),
  - [2..] — локальные переменные.
- **Стек catch** — хранит записи: (метка, адрес конца блока, номер кадра, SP).

**Регистры**:
- ``PC`` — программный счётчик (адрес текущей инструкции).
- ``ACC`` — аккумулятор (результат последней операции).
- ``FRAME`` — текущий кадр активации.
- ``SP`` — указатель стека данных.
- ``CATCH-SP`` — указатель стека catch.

**Сборка мусора**:
- **Фаза пометки**: обход от корневых объектов — ACC, FRAME, стек (от SP), глобальные переменные, константы.
- **Фаза очистки**: удаление непомеченных объектов.
